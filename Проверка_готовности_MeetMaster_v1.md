| Документ | Статус |

## Где владелец проекта нужен обязательно

Без этих данных невозможно запустить реального бота и реальную интеграцию с Google Календарем.

| Что нужно | Зачем | Как передавать |
|---|---|---|
| Токен Telegram-бота | чтобы бот работал в Telegram | не отправлять в обычный чат |
| Telegram ID администратора | чтобы защитить админку | можно передать числом, это не пароль |
| Google-аккаунт календаря | чтобы встречи попадали в правильный календарь | владелец входит сам через OAuth |
| Разрешение Google OAuth | чтобы бот мог читать и создавать события | владелец подтверждает доступ в браузере |
| Доступ к серверу | чтобы развернуть бота | безопасным способом |
| Данные PostgreSQL | чтобы бот подключался к базе | через `.env` на сервере |

## Минимальное участие владельца проекта

Чтобы сделать бот максимально без вас, достаточно 5 точек участия.

| № | Когда | Что нужно сделать владельцу |
|---:|---|---|
| 1 | Перед разработкой | создать Telegram-бота через BotFather или дать разработчику возможность это сделать |
| 2 | Перед разработкой | дать Telegram ID администратора |
| 3 | При настройке Google | войти в Google-аккаунт и разрешить доступ к календарю |
| 4 | При запуске | дать доступ к серверу или выполнить инструкцию разработчика |
| 5 | Перед сдачей | пройти финальный тест: создать заявку, подтвердить, проверить Google Календарь |

Все остальное разработчик должен проверять сам.

## Что желательно добавить в разработку, чтобы я мог проверять максимум сам

Эти пункты нужно включить в работу разработчика. Они снизят зависимость от ручной проверки.

| Пункт | Зачем |
|---|---|
| Тестовый режим | можно проверять встречи и напоминания без реальных клиентов |
| Тестовый администратор | можно проверять админку на отдельном Telegram ID |
| Тестовый Google Календарь | можно проверять интеграцию без риска испортить основной календарь |
| Команда `/health` для администратора | быстро понять, что бот, база и Google доступны |
| Команда `/admin_stats` | увидеть количество заявок и ошибок |
| Команда `/admin_settings` | проверить рабочие часы, буфер и закрытые дни |
| Лог ошибок без секретов | разработчик видит проблему, не раскрывая токены |
| Автотесты расчета слотов | проверяют выходные, буфер, занятость, закрытые дни |

## Команды администратора для MVP

Чтобы ботом было удобно управлять без веб-кабинета, в MVP нужны такие админ-команды.

| Команда | Назначение |
|---|---|
| `/admin` | открыть админ-меню |
| `/admin_requests` | показать заявки на согласовании |
| `/admin_settings` | показать настройки записи |
| `/admin_close_day` | закрыть дату для записи |
| `/admin_open_day` | открыть закрытую дату |
| `/admin_stats` | показать простую статистику |
| `/health` | проверить состояние бота, базы и Google |

В обычном пользовательском меню эти команды не показываются.

## Обязательные автоматические проверки

Разработчик должен добавить автоматические проверки хотя бы для ключевых правил.

| Проверка | Что проверяет |
|---|---|
| Рабочие дни | суббота и воскресенье недоступны |
| Рабочее время | слоты только с 09:00 до 17:00 |
| Минимальный срок записи | нельзя записаться меньше чем за 1 день |
| Буфер | между встречами есть 15 минут |
| Длительность | доступны 30, 45, 60 минут |
| Закрытый день | закрытая дата не показывается |
| Блокировка слота | заявка на согласовании скрывает слот |
| Email | некорректный email не принимается |
| Статусы | заявка корректно меняет статус |
| Права доступа | обычный пользователь не может открыть админку |

## Что я должен проверить сам при разработке

Если я буду делать бота, я должен сам проверить:

| Блок | Проверка |
|---|---|
| Telegram | `/start`, меню, кнопки, пользовательский сценарий |
| Админка | подтверждение, отклонение, перенос, отмена |
| Слоты | рабочие дни, рабочее время, буфер, закрытые дни |
| База | сохранение пользователей, заявок, встреч, настроек |
| Google | чтение занятости, создание, перенос, отмена события |
| Уведомления | все статусы и напоминание за 1 час |
| Безопасность | админка только для администратора |
| Логи | ошибки пишутся, секреты не выводятся |
| Сервер | бот работает после перезапуска |
| Бэкап | резервная копия PostgreSQL создается |

## Что владелец проекта проверяет в финале

Финальная проверка без знания кода:

1. Открыть Telegram-бота.
2. Нажать `/start`.
3. Нажать "Записаться на встречу".
4. Выбрать рабочий день.
5. Выбрать длительность 30 минут.
6. Выбрать свободное время.
7. Ввести email.
8. Ввести описание "Тестовая встреча".
9. Отправить заявку.
10. Открыть админское меню.
11. Подтвердить заявку.
12. Открыть Google Календарь.
13. Проверить, что встреча появилась.
14. Проверить, что email добавлен в участники.
15. Отменить тестовую встречу.
16. Проверить, что встреча пропала или стала отмененной в Google Календаре.

Если все пункты выполнены, MVP можно принимать.

## Потенциальные пробелы и как их закрыть

| Пробел | Что делаем |
|---|---|
| Не указан точный Telegram для помощи | оставить текст "Связаться с администратором" |
| Не указан конкретный Google-календарь | использовать основной календарь подключенного Google-аккаунта |
| Не указан способ хранения бэкапов | хранить на сервере и согласовать отдельную папку при настройке |
| Не указан текст всех уведомлений | разработчик готовит базовые тексты на русском, владелец правит только если нужно |
| Не указан дизайн MiniApp | не нужен для MVP |

## Вывод

Текущий пакет документов достаточен, чтобы начать разработку MVP.

Чтобы сделать бота максимально автономно, разработчику не хватает только реальных доступов:

| Доступ | Без него можно писать код | Без него можно запустить реальный MVP |
|---|---|---|
| Telegram Bot Token | да | нет |
| Telegram ID администратора | частично | нет |
| Google OAuth | частично | нет |
| Сервер | локально да | на продакшене нет |
| PostgreSQL на сервере | локально да | на продакшене нет |

Все секреты и пароли должны передаваться безопасно и не должны попадать в обычные текстовые файлы, чат или логи.
