# Архитектура и технологии MeetMaster v1

## Назначение документа

Этот документ фиксирует архитектуру и технологии для разработки Telegram-бота MeetMaster.

Документ нужен, чтобы разработчик понимал:

- из каких частей должен состоять проект;
- какие технологии использовать;
- что входит в MVP;
- что нужно заложить на будущее;
- как проверять, что архитектура реализована правильно.

---

## 1. Краткое решение

Для MVP используется простая и недорогая архитектура:

| Компонент | Решение |
|---|---|
| Язык разработки | Python |
| Telegram-бот | aiogram |
| База данных | PostgreSQL |
| Интеграция с календарем | Google Calendar API |
| Авторизация Google | Google OAuth |
| Админка | внутри Telegram |
| Напоминания | встроенный планировщик задач |
| Запуск бота | polling |
| Сервер | существующий сервер владельца проекта |
| MiniApp | не входит в MVP, заложить возможность добавить в v2 |

---

## 2. Почему выбран такой вариант

| Решение | Причина |
|---|---|
| Python | популярный язык для Telegram-ботов, проще найти разработчика |
| aiogram | современная библиотека для Telegram-ботов на Python |
| PostgreSQL | надежная бесплатная база, подходит для роста проекта |
| Google Calendar API | официальный способ работать с Google Календарем |
| Админка в Telegram | дешевле и быстрее, чем отдельный веб-кабинет |
| Polling | проще для MVP, можно запустить без домена и SSL |
| Существующий сервер | снижает расходы на инфраструктуру |

---

## 3. Общая схема работы

```text
Пользователь
  |
  | Telegram
  v
Telegram-бот MeetMaster
  |
  | сохраняет заявки, пользователей, настройки
  v
PostgreSQL
  |
  | проверяет занятость / создает встречу
  v
Google Calendar API

Администратор работает с заявками через Telegram-бота.
```

## 4. Основные блоки системы

| Блок | Назначение |
|---|---|
| Пользовательский блок | запись на встречу, выбор даты, времени, email и описания |
| Админский блок | подтверждение, отклонение, перенос, отмена, закрытие дней |
| Блок расчета слотов | расчет свободного времени с учетом правил |
| Блок Google Календаря | чтение занятости и создание событий |
| Блок базы данных | хранение пользователей, заявок, встреч, настроек |
| Блок уведомлений | статусы заявки, отмены, переносы, напоминания |
| Блок логирования | запись ошибок и технических событий |
| Блок настроек | хранение рабочих часов, буфера, закрытых дней |

## 5. Структура проекта

```text
meetmaster/

    services/
```

### Расшифровка структуры простыми словами

| Папка или файл | Что там должно быть |
|---|---|
| `app/bot` | все, что связано с Telegram: команды, кнопки, сообщения |
| `app/services` | основная логика записи, согласования, отмены и переноса |
| `app/integrations` | подключение к Google Календарю |
| `app/db` | база данных, таблицы, работа с данными |
| `app/scheduler` | напоминания за 1 час до встречи |
| `app/config` | настройки проекта |
| `tests` | автоматические проверки |
| `docker-compose.yml` | запуск базы и приложения через Docker, если это удобно |
| `.env.example` | пример файла настроек без реальных секретов |
| `README.md` | инструкция по запуску и использованию |

## 6. Хранение данных

База данных: PostgreSQL.

### Основные таблицы

| Таблица | Что хранит |
|---|---|
| `users` | пользователей Telegram |
| `booking_requests` | заявки на встречи |
| `meetings` | подтвержденные встречи и связь с Google Calendar |
| `settings` | настройки записи |
| `closed_days` | дни, закрытые администратором |
| `status_history` | история изменения статусов |

### Что обязательно хранить

| Сущность | Данные |
|---|---|
| Пользователь | Telegram ID, имя, фамилия, username, email |
| Заявка | дата, время, длительность, email, описание, статус |
| Встреча | ID события Google Calendar, статус, дата создания |
| Настройки | рабочие часы, буфер, минимальный срок записи |
| Закрытый день | дата, причина, дата создания |

## 7. Интеграция с Telegram

Используется Telegram Bot API через библиотеку aiogram.

### Требования

| Требование | Описание |
|---|---|
| Главное меню | кнопки "Записаться на встречу", "Мои заявки", "Помощь" |
| Админские кнопки | подтвердить, отклонить, перенести, отменить |
| Защита админки | доступ только по Telegram ID администратора |
| Русский язык | все сообщения пользователю на русском |
| Обработка ошибок | пользователь не должен видеть технические ошибки |

---

## 8. Интеграция с Google Календарем

Используется Google Calendar API.

### Что должен делать бот

| Действие | Описание |
|---|---|
| Читать занятость | получать занятые интервалы из календаря |
| Показывать свободные слоты | исключать занятое время |
| Создавать событие | после подтверждения заявки администратором |
| Добавлять участника | добавлять email пользователя в событие |
| Переносить событие | менять дату и время при переносе |
| Отменять событие | удалять или отменять событие при отмене встречи |

### Авторизация

Google-аккаунт администратора подключается через Google OAuth.

Доступ должен быть только к календарю, который используется для встреч.

---

## 9. Расчет свободных слотов

Бот должен учитывать:

| Правило | Значение |
|---|---|
| Рабочие дни | понедельник-пятница |
| Нерабочие дни | суббота, воскресенье |
| Рабочее время | 09:00-17:00 |
| Минимальный срок записи | 1 день |
| Буфер между встречами | 15 минут |
| Длительность | 30, 45, 60 минут |
| Закрытые дни | даты, закрытые администратором |
| Занятость календаря | занятые интервалы Google Календаря |
| Заявки на согласовании | временно блокируют слот |

Перед созданием события в Google Календаре бот должен повторно проверить, что слот свободен.

Для MVP используется встроенный планировщик задач.

### Требования

## 11. Запуск на сервере

Бот запускается на существующем сервере владельца проекта.

### Рекомендуемый вариант запуска MVP

| Параметр | Решение |
|---|---|
| Режим Telegram | polling |
| Причина | проще для запуска, не нужен домен и SSL |
| База | PostgreSQL на сервере |
| Автозапуск | системный сервис или Docker |
| Логи | файлы логов или системные логи сервера |
| Бэкапы | ежедневная резервная копия PostgreSQL |

### Почему polling

Polling означает, что бот сам регулярно спрашивает Telegram, есть ли новые сообщения.

Для MVP это удобно, потому что:

- не нужен домен;
- не нужен SSL-сертификат;
- проще запуск на сервере;
- дешевле внедрение.

В будущем можно перейти на webhook, если появится MiniApp, веб-кабинет или высокая нагрузка.

## 12. Безопасность

| Требование | Описание |
|---|---|
| Токены и пароли | хранить только в переменных окружения |
| `.env` | не добавлять в репозиторий |
| `.env.example` | хранить только пример без реальных секретов |
| Админ-доступ | проверять Telegram ID администратора |
| Google OAuth | не хранить лишние права доступа |
| Логи | не выводить токены, пароли и ключи |
| База | доступ только с сервера или по защищенному соединению |

---

## 13. Логи и бэкапы

### Логи

В логах должны фиксироваться:

| Событие | Нужно логировать |
|---|---|
| Ошибки Telegram | да |
| Ошибки Google Calendar API | да |
| Ошибки базы данных | да |
| Создание заявки | да |
| Изменение статуса заявки | да |
| Отмена или перенос встречи | да |

В логах не должно быть токенов, паролей, ключей и приватных данных сверх необходимого.

### Бэкапы

| Объект | Требование |
|---|---|
| PostgreSQL | ежедневный бэкап |
| Проверка бэкапа | разработчик должен проверить восстановление на тестовой базе |
| Хранение | место хранения согласуется при настройке сервера |

## 14. MiniApp в будущем

MiniApp не входит в MVP.

При этом архитектура должна позволять добавить MiniApp без полной переделки проекта.

### Что нужно заложить сейчас

| Что заложить | Зачем |
|---|---|
| Отдельная бизнес-логика в `services` | чтобы MiniApp могла использовать ту же логику |
| Отдельный расчет слотов | чтобы не переписывать календарь |
| База PostgreSQL | чтобы данные подходили для чат-бота и MiniApp |
| Четкие сущности заявок и встреч | чтобы MiniApp показывала те же данные |

## 15. Проверка архитектуры

### Что разработчик обязан проверить сам

| Проверка | Как проверяет |
|---|---|
| Проект запускается | запускает приложение на сервере или локально |
| Бот отвечает | отправляет `/start` |
| База подключается | создает тестовую заявку |
| Google Calendar подключается | получает занятость календаря |
| Настройки читаются из `.env` | проверяет запуск без вывода секретов |
| Админ-доступ защищен | проверяет обычным Telegram-аккаунтом |
| Логи пишутся | вызывает тестовую ошибку |

### Что проверяет владелец проекта

| Проверка | Как проверить |
|---|---|
| Бот открывается | открыть бота в Telegram |
| Главное меню есть | нажать `/start` |
| Запись запускается | нажать "Записаться на встречу" |
| Заявка доходит до администратора | создать тестовую заявку |
| Встреча появляется в Google Календаре | подтвердить заявку и открыть календарь |
