# План самостоятельной разработки Telegram Mini App для MeetMaster

Дата: 2026-05-14

## Цель

Сделать Telegram Mini App как дополнение к уже работающему боту MeetMaster из урока `6-4`, не ломая текущего бота и не заменяя его.

Mini App должен дать пользователю удобный визуальный интерфейс для записи на встречу: календарь, выбор длительности, выбор свободного времени, ввод email и описания, отправка заявки, просмотр своих заявок.

Текущий бот должен продолжить работать как сейчас.

## Главный принцип работы

Я буду делать Mini App рядом с существующим ботом, а не вместо него.

Рабочий бот остается запасным и стабильным способом записи. Все новые части сначала проверяются отдельно, потом подключаются к боту одной кнопкой.

## Что я не буду делать без необходимости

Я не буду:

- переписывать существующего бота с нуля;
- удалять текущие обработчики aiogram;
- менять работающий сценарий записи через сообщения;
- трогать `.env`, токены и Google credentials без необходимости;
- удалять данные на сервере;
- удалять Supabase и n8n на VPS;
- запускать опасную очистку Docker без анализа;
- выкладывать секреты в GitHub.

## Что уже есть и будет использовано

В текущем проекте уже есть:

- Telegram-бот на aiogram;
- PostgreSQL;
- Docker Compose;
- Alembic-миграции;
- модели пользователей, заявок, встреч, закрытых дней, настроек и истории статусов;
- расчет свободных дат и слотов;
- проверка email;
- админское согласование заявок;
- перенос и отмена встреч;
- интеграция с Google Calendar;
- уведомления пользователю и администратору.

Эту логику нужно использовать повторно, а не писать второй набор правил.

## Общая архитектура

Планируемая структура:

```text
meetmaster/
  app/                  существующий бот и общая логика
  mini_app/
    backend/            API для Telegram Mini App
    frontend/           интерфейс Mini App
```

Backend:

- FastAPI;
- использует существующие модели, репозитории и сервисы;
- подключается к той же PostgreSQL;
- проверяет Telegram WebApp initData;
- создает заявки в существующей базе;
- уведомляет администратора через Telegram Bot API.

Frontend:

- React + Vite;
- Telegram Web Apps SDK;
- мобильный интерфейс внутри Telegram;
- календарь, слоты, форма заявки, список заявок.


4. Подготовлю commit с текущим стабильным состоянием.
5. После commit можно будет безопасно начинать новую разработку.

Результат этапа:

- есть точка возврата;
- текущий бот не потерян;
- можно спокойно добавлять Mini App.

## Этап 2. Создать отдельную ветку или рабочую копию

Что я сделаю:

1. Создам отдельную ветку для Mini App, например:

```bash

```

2. Если нужно работать в папке урока `6-7`, аккуратно перенесу туда копию проекта или создам новый проект, который будет ссылаться на существующую архитектуру.

Рекомендуемый вариант: ветка в Git, потому что так проще не потерять историю и откатиться.

Результат этапа:

- текущий бот в основной ветке не ломается;
- Mini App разрабатывается отдельно.

## Этап 3. Выделить общую бизнес-логику

Сейчас часть логики находится прямо в обработчиках бота. Для Mini App нужно сделать так, чтобы бот и Mini App использовали одну и ту же логику.

Что я вынесу или оформлю:

- получение доступных дат;
- получение свободных слотов;
- создание заявки;
- получение заявок пользователя;
- уведомление администраторов;
- проверку прав администратора;
- изменение статусов заявок.

Пример будущих сервисов:

```text
app/services/booking_flow_service.py
app/services/admin_action_service.py
app/services/telegram_auth_service.py
```

Результат этапа:

- логика не дублируется;
- старый бот и Mini App работают по одним правилам;
- меньше риск расхождения поведения.

## Этап 4. Добавить backend для Mini App

Что я сделаю:

1. Добавлю FastAPI-зависимость в `pyproject.toml`.
2. Создам backend Mini App.
3. Подключу существующую базу данных.
4. Добавлю проверку Telegram `initData`.
5. Реализую API.

Минимальные endpoints:

```text
GET  /api/me
GET  /api/booking/dates
GET  /api/booking/durations
GET  /api/booking/slots?date=YYYY-MM-DD&duration=45
POST /api/booking/requests
GET  /api/booking/my-requests
```

Формат создания заявки:

```json
{
  "date": "2026-05-19",
  "time": "09:45",
  "duration_minutes": 45,
  "email": "client@example.com",
  "description": "Описание встречи"
}
```

Что backend должен сделать при создании заявки:

1. Проверить пользователя через Telegram initData.
2. Проверить email.
3. Проверить дату и слот.
4. Проверить, что слот все еще свободен.
5. Создать или обновить пользователя.
6. Создать заявку со статусом `pending_approval`.
7. Записать историю статуса.
8. Отправить сообщение администраторам.
9. Вернуть результат frontend.

Результат этапа:

- Mini App получает безопасный API;
- frontend не имеет доступа к базе и секретам.

## Этап 5. Добавить тесты backend

Что я проверю тестами:

- большой Telegram ID проходит корректно;
- пользователь видит только свои заявки;
- нельзя создать заявку на занятый слот;
- нельзя создать заявку на закрытый день;
- нельзя создать заявку с плохим email;
- нельзя создать заявку без валидного Telegram initData;
- админские данные не доступны обычному пользователю.

Результат этапа:

- backend можно менять без страха;
- основные ошибки ловятся до сервера.

## Этап 6. Создать frontend Mini App

Что я сделаю:

1. Создам React + Vite приложение.
2. Подключу Telegram Web Apps SDK.
3. Сделаю мобильный интерфейс.
4. Добавлю экраны:

```text
Главный экран

Выбор длительности
Выбор времени
Форма email и описания
Проверка заявки
Успешная отправка
```

Требования к интерфейсу:

- без лишнего текста;
- понятные статусы заявок;

- адаптация под Telegram theme params.

Результат этапа:

- пользователь может пройти весь сценарий записи внутри Telegram Mini App.

## Этап 7. Связать Mini App с ботом

Что я сделаю:

4. Проверю, что старый сценарий через сообщения работает.

Результат этапа:

## Этап 9. Проверка внутри Telegram

Для настоящей проверки Telegram Mini App нужен HTTPS URL.

Варианты:

1. Временный туннель для теста.
2. Домен на сервере.
3. Поддомен для Mini App.

Что я сделаю:

1. Подниму frontend и backend на временном HTTPS URL.
2. Укажу URL в кнопке бота.
3. Открою Mini App из Telegram.
4. Проверю полный путь заявки.

Результат этапа:

- Mini App реально открывается внутри Telegram;
- Telegram initData приходит и проверяется.

## Этап 10. Размещение на сервере

На сервере уже есть:

```text
/opt/stack/supabase
/opt/stack/n8n
/opt/meetmaster/app
```

Из-за малого места на диске сначала проверю:

```bash
df -h /
docker system df
du -h --max-depth=1 /var/lib/docker 2>/dev/null | sort -h
```

Потом добавлю сервисы аккуратно:

```text
app-bot-1       существующий бот
app-db-1        существующая база MeetMaster
mini-api        backend Mini App
mini-web        frontend Mini App или статическая сборка
reverse-proxy   HTTPS-прокси, если нужен
```

Что важно:

- не удалять существующие сервисы;
- не менять Supabase и n8n;
- не публиковать Postgres наружу;
- не хранить секреты в образе;
- использовать `.env` на сервере.

Результат этапа:

- Mini App работает на сервере;
- бот продолжает работать;
- заявка создается в той же базе.

## Этап 13. Откат, если что-то пойдет не так

Если Mini App сломается, старый бот должен продолжить работать.

План отката:

1. Убрать кнопку Mini App из меню бота.
2. Остановить только сервисы Mini App.
3. Оставить `app-bot-1` и `app-db-1` работающими.
4. Вернуться к стабильному commit.

Команды на сервере будут зависеть от итогового compose-файла, но принцип такой:

```bash
cd /opt/meetmaster/app
docker compose stop mini-api mini-web
```

Существующий бот при этом не должен останавливаться.
