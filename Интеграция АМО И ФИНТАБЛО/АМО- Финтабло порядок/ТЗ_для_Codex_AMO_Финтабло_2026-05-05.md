## 2. Рабочая схема workflow

Порядок узлов должен быть строго таким:

```text
Webhook
  -> If
  -> Проверить и запомнить ID сделки AMO
  -> HTTP Request
```

Что делает каждый узел:

| Узел | Назначение |
|---|---|
| Webhook | принимает событие из AMO |
| If | пропускает только status_id = 142 |
| Проверить и запомнить ID сделки AMO | отсекает повтор по ID сделки AMO |
| HTTP Request | создает сделку в Финтабло |

## 3. Ключевая ошибка, которую нашли

Ошибка была в узле If.

Было:

```text
={{ $json["body"]["leads[update][0][status_id]"] }}


```

Из-за лишних переносов строк n8n сравнивал не чистое значение `142`, поэтому событие уходило в False Branch.

Должно быть:

```text
={{ $json["body"]["leads[update][0][status_id]"] }}
```

Правое значение:

```text
142
```

Итог: если AMO присылает `status_id = 142`, сделка должна идти дальше в Финтабло.

## 4. Антидубль

Антидубль хранит уже отправленные AMO ID в static data workflow.

Логика:

| Условие | Действие |
|---|---|
| ID сделки AMO уже есть в памяти | не отправлять в Финтабло |
| ID сделки AMO новый | запомнить ID и отправить в Финтабло |

Узел:

```text
Проверить и запомнить ID сделки AMO
```

Память антидубля на момент фиксации содержит:

```text
26141561
26141713
26141887
26152837
26153041
```

Эти ID уже не должны повторно создавать сделки в Финтабло.

|---|---:|---|

Итого по найденным ID:

```text

```

После проверки в интерфейсе Финтабло итог стал:

```text
6 сделок на 1 135 000 руб.
```

Значит одна сделка на 5 000 руб. была в Финтабло/AMO вне найденной истории n8n или пришла отдельно. В интерфейсе итог сходится.

## 6. Безопасность секретов

Нельзя выводить:

```text
Authorization
Bearer
API token
Telegram token
OpenRouter token
.env
```

Если токен случайно вывели в лог или чат:

1. Не повторять токен в ответе.
2. Написать `[СКРЫТО]`.
3. Предупредить, что токен нужно сменить.
4. Перенести токен в n8n Credentials, а не хранить в параметрах узла.

Токен Финтабло был найден прямо в HTTP Request. Его надо считать раскрытым и заменить.

## 8. Если сделка не попала в Финтабло

Порядок:

1. Не менять workflow вслепую.
2. Открыть последние execution в n8n.
3. Проверить, дошел ли запуск до HTTP Request.
4. Если дошел до Webhook и If, но не дошел до HTTP Request, смотреть If.
5. Если дошел до HTTP Request, смотреть ответ Финтабло.
6. Если Финтабло вернул 200, искать сделку в Финтабло по комментарию `ID сделки: <AMO_ID>`.

Коммент в Финтабло должен быть:

```text
Создано из amoCRM. ID сделки: <AMO_ID>
```

## 9. Если появились дубли

Порядок:

1. Найти AMO ID в комментариях Финтабло.
2. Проверить, есть ли этот ID в staticData workflow.
3. Проверить, что в AMO нет второго webhook на cloud n8n.
4. Проверить, что cloud workflow AMO -> Финтабло выключен.
5. Не удалять сделки автоматически. Сначала показать список дублей пользователю.

## 10. Что нельзя делать

Нельзя:

- возвращаться к cloud n8n для этой интеграции;
- менять URL webhook в AMO без проверки;
- снимать с публикации рабочий workflow;
- удалять execution history;
- удалять staticData антидубля;
- публиковать Telegram workflow вместо AMO workflow;
- выводить токены и .env в чат;
- запускать тест со status_id = 142 без понимания, что он создаст сделку в Финтабло.

## 11. Резервные копии

Перед каждой правкой экспортировать workflow:

```bash
cd /opt/stack/n8n
sudo docker exec n8n n8n export:workflow --id=9sjdTrPk9C33LWPB --output=/tmp/amo_fintablo_backup.json
sudo docker cp n8n:/tmp/amo_fintablo_backup.json /opt/stack/backups/n8n-workflows/amo_fintablo_backup.json
```

После импорта n8n может деактивировать workflow. Нужно включить обратно:

```bash
sudo docker exec n8n n8n update:workflow --id=9sjdTrPk9C33LWPB --active=true
sudo docker compose --env-file .env -f compose.yml restart n8n
```

Потом снова проверить webhook и healthz.
