Памятка: серверный n8n + AMO + Финтабло

Дата фиксации: 2026-05-05

Рабочий курс:
- работаем только с серверным n8n;
- к n8n cloud не возвращаемся;
- причина разбора: из AMO дублировались сделки в Финтабло;
- задача: держать рабочую серверную версию без повторного появления дублей.

Проверено на сервере 2026-05-05:
- в серверном n8n есть 1 workflow;
- workflow называется "amoCRM → Финтабло: завершенные сделки";
- workflow активен;
- production webhook зарегистрирован;
- схема узлов: Webhook -> If -> HTTP Request;
- HTTP Request в Финтабло не стоит напрямую после Webhook;
- условие If пропускает дальше только AMO update со status_id = 142;
- HTTP Request отправляет POST на api.fintablo.ru;
- последние webhook-запуски завершены со статусом success.

Логика исправления, которая фактически видна в workflow:
- AMO может присылать несколько update-событий по одной сделке;
- серверный n8n отправляет в Финтабло не каждый update, а только update с нужным статусом 142;
- за счет этого обычные промежуточные обновления AMO не создают повторные записи в Финтабло.

Что не видно в текущей схеме:
- отдельной постоянной таблицы/хранилища обработанных ID сделок в workflow не видно;
- если AMO повторно пришлет именно status_id = 142 по той же сделке, нужна дополнительная защита по ID сделки AMO.

Контрольные точки:
- workflow в серверном n8n должен быть активен;
- AMO должен смотреть на production webhook серверного n8n;
- старый cloud workflow не должен параллельно создавать записи в Финтабло;
- test webhook / webhook-test не использовать в рабочей интеграции;
- после правки проверять одну тестовую сделку AMO и смотреть, что в Финтабло появляется ровно одна запись.

Секреты:
- пароли, токены, API-ключи и значения .env в этот файл не записывать.
